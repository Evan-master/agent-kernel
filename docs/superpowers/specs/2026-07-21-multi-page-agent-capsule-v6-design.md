# Multi-page Agent Capsule V6 Design

Status: Implemented and validated on 2026-07-21

## Objective

Remove the one-page ceiling from native Agent software while preserving the
bounded, allocation-free Capsule parser and deterministic physical ownership
model. V6 gives every isolated Agent a fixed four-page executable window,
loads one verified flat code image across those pages, and proves execution
after the first 4 KiB boundary with the ring-3 Resource Manager.

## Ownership Boundaries

- `agent-kernel-x86_64::agent_image` owns the unchanged 32-byte Capsule v1
  header, exact raw-image digest, declared code length, entry offset, and the
  new 16 KiB code bound.
- `UserMemoryLayout` owns one contiguous four-page virtual code window and the
  shifted fixed locations of every non-code Agent page.
- `PreparedAgentMemory` owns four physical code frames, deterministic zero
  padding, page-spanning copy verification, restart retention, and frame-index
  discipline.
- The private page-table adapter maps every code frame user-readable and
  executable with `WRITABLE` and `NO_EXECUTE` clear, while the kernel root
  excludes the complete Agent region.
- `AgentMemoryIdentity` and the reclamation pool own all four code frames
  through teardown, zeroing, reuse, cancellation, and rollback.
- Boot Capsule modules own exact generated bytes, SHA-256 values, operation
  transcripts, and return offsets.

No variable allocator, writable code mapping, runtime relocation, arbitrary
userspace address, or host loader enters the kernel execution path.

## Capsule Contract

Capsule v1 remains byte-compatible:

| Offset | Bytes | Field |
| ---: | ---: | --- |
| `0` | 8 | magic: `AGNTIMG\0` |
| `8` | 2 | format version: `1` |
| `10` | 2 | architecture: x86_64 (`1`) |
| `12` | 2 | image kind |
| `14` | 2 | flags: zero |
| `16` | 2 | ABI version |
| `18` | 2 | entry version |
| `20` | 4 | entry offset |
| `24` | 4 | declared code length: `1..16,384` |
| `28` | 4 | reserved: zero |
| `32` | N | exact code bytes |

The parser still requires `entry_offset < code_length` and exact raw length.
`VerifiedAgentImage` still binds the complete header and declared code bytes to
the verified kernel record through SHA-256. Unused bytes in the final physical
code page are kernel-zeroed and immutable.

## Fixed Virtual Layout

`AGENT_REGION_BASE` remains `0x0000400000000000`.

| Region | Start | Pages | Ring-3 policy |
| --- | ---: | ---: | --- |
| Code | `0x0000400000000000` | 4 | user RX |
| Signal | `0x0000400000004000` | 1 | user R, NX |
| Guard | `0x0000400000005000` | 1 | unmapped |
| Stack | `0x0000400000006000` | 4 | user RW, NX |
| Lazy data | `0x000040000000a000` | 1 | initially unmapped |
| Runtime page | `0x000040000000b000` | 1 | initially unmapped |
| Runtime region | `0x000040000000c000` | 8 | initially unmapped |
| Call data | `0x0000400000014000` | 1 | user RW, NX |

All addresses remain in the dedicated Agent P4 slot. Existing Agent Call IDs,
register contracts, Event semantics, and Capsule header bytes remain stable.
The fixed user virtual addresses after code are an architecture-contract change
and every native proof image is regenerated in the same milestone.

## Physical Identity

The content-frame order becomes:

```text
0..3   code pages
4      signal page
5..8   stack pages
9      lazy-data backing page
10     call-data page
```

With four private page-table frames, one Agent owns 15 frames. Identity
construction still rejects aliases and the reclamation pool still transfers a
whole identity atomically.

## Loading And Mapping

Preparation allocates all four code frames before any page-table install. It
zeros every page, copies each declared code chunk into the matching physical
page, and reads every declared byte back through the supervisor physical
window. Every code page is then mapped at `code_start + page * 4096` with
`PRESENT | USER_ACCESSIBLE`; writable, NX, and huge-page bits are rejected by
validation.

Reused identities must be fully zero before loading. Address-space restart
clears signal, stack, lazy-data, and call-data frames while retaining all four
immutable code frames. Final reclamation clears and verifies every content and
page-table frame before pool commit.

## Native Cross-page Proof

The Resource Manager keeps Calls 1 through 52 and adds 64 deterministic NOP
bytes before its final result and completion calls. Its code length therefore
exceeds 4 KiB, while the final two interrupt return offsets land in code page
two. QEMU completion proves:

- parser acceptance and full-Capsule digest binding;
- four physical code frames and four RX leaf mappings;
- instruction fetch across the first page boundary;
- correct Agent Call return-offset capture beyond `4096`;
- unchanged Resource Manager operations and semantic Event transcript;
- complete 15-frame reclamation and reuse paths.

The runtime emits `AGENT_KERNEL_NATIVE_MULTI_PAGE_CAPSULE_OK` only after the
cross-page transcript and final Store evidence agree.

## Verification Gates

- parser accepts code lengths `4096`, `4097`, and `16384`, and rejects `16385`;
- entry offsets on both sides of a page boundary remain digest-bound;
- layout contracts prove all fixed addresses, code containment, and one P4
  slot;
- identity, reclamation, cancellation, reuse, and admission contracts prove 11
  content frames and 15 owned frames;
- page-table install and validation cover all four RX pages;
- Resource Manager assembly, generated Capsule, SHA-256, return offsets, and
  Release ELF occurrences agree byte-for-byte;
- focused and Workspace tests, Supervisor simulation, `no_std`, strict Clippy,
  debug/release QEMU, and final source scans pass.
